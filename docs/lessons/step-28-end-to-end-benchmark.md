# Step 28：端到端 benchmark vs pak（终于敢同台）

> 模块：L Benchmark ｜ 对应提交：本步提交（commit 信息含 `Lesson: docs/lessons/step-28-end-to-end-benchmark.md`）｜ 产物：`scripts/bench.sh` + `BENCHMARK.md` ｜ 上一步：[Step 27](step-27-warm-cache.md)

## 0. 一句话目标
把 uvr 与 pak 做一次**端到端、多轴、诚实**的对照：解析延迟、暖缓存 lock、以及完整的"装一个包"。

## 1. 前置回顾
此前我们已补齐 多仓库（H）+ 暖缓存（I），加上既有的 求解 / 下载 / 安装——uvr 终于具备和 pak"全流程"同台的能力。本步把对比做出来。

## 2. "测试"：可复现的真实数字
benchmark 的"绿"= 脚本稳定跑出可复现的真实数字。`scripts/bench.sh` 新增端到端安装段（uvr cold/warm vs pak warm，装进临时库）。

## 3. 实现到通过（真实数字）
关键点：
- **公平选包**：用纯 R 包 `dotenv`（无编译）——两边都不编译，安装这块才公平。
- **装进临时库**：`--lib /tmp/...`（uvr）与 `pak::pkg_install(lib=...)`（pak），都不碰全局环境。
- **冷/暖**：uvr 清空 `.uvr-cache` 测冷、保留测暖；pak 用其自带缓存（warm）。

结果（本机，min/single）：

| 轴 | uvr | pak |
|---|---|---|
| 解析延迟（一次性） | **6 ms** | 5415 ms |
| lock 暖缓存 | 冷 165 → **暖 6 ms** | — |
| 端到端安装 dotenv | 冷 2080 / **暖 1575 ms** | 5478 ms |

即便 uvr **冷** 也比 pak **暖** 快 ~2.6×。

## 4. 改了哪些文件 / 加了什么
- `scripts/bench.sh`：新增端到端安装对照段（uvr cold/warm vs pak warm）。
- `BENCHMARK.md`：重写为多轴报告 + 诚实解读 + 待补齐项。

## 5. 学到的语法 / 技巧
- **`${VAR}` 显式界定**：`echo "...${PKG}（..."` —— 变量名后紧跟多字节字符（全角括号）时，bash 3.2 会把字节误并进变量名；用 `${PKG}` 划清边界。（这一步真踩了这个坑并修掉。）
- **单次 vs 多次计时**：慢且有副作用的安装用单次（`once_ms`）；快操作用多次取最小（`min_ms`）。
- **隔离到临时库 / 临时缓存**：测安装不污染真实环境。

## 6. 设计巧思 / 方法论
- **公平选轴、公平选样本**：只比能公平比的（纯 R 包、暖缓存），把不公平的（含编译包、binary）**显式排除并说明**。
- **共享地板要点名**：`R CMD INSTALL` 的 ~1.5s 是两边共担的物理地板；uvr 的领先来自"安装之外"的开销，而非装得更快——这话写进报告，避免误导。

## 7. 领域知识
- **pak 的成本结构**：R 启动（267ms）+ 每进程重载元数据库（数秒）+ 安装。它为"R 会话内反复用"而设计，一次性 CLI 调用是它的最坏情形。
- **uvr 的成本结构**：原生启动（~ms）+ 暖缓存读盘 + `R CMD INSTALL`。开销几乎只剩那块共享地板。
- **binary vs source**：pak 用预编译 binary 免编译；uvr 目前装源码。纯 R 包没差别，含编译的包 pak 更快——这是 uvr 的下一块拼图。

## 8. 软件设计理念 / 科研诚信
- **诚实报告**：赢的说赢、平的说平、还不公平的明说不公平并划定范围。可复现脚本入库，任何人可复算。
- **以终为始**：为了"能和 pak 公平对比"这个目标，反推需要补齐的能力（多仓库→缓存→…），逐块建造——这正是 uvr 这一路的主线。

## 9. 小测验（自测）
1. 为什么 benchmark 选纯 R 包 `dotenv`，而不是一个含 C 代码的大包？
2. uvr 端到端安装比 pak 快，主要快在哪里？哪部分其实打平？
3. `${PKG}` 与 `$PKG` 在"后接全角括号"时有何不同？
4. 报告里为什么要专门写"binary 包未支持、含编译包 pak 更快"？

## 10. 参考答案
1. 纯 R 包两边都不编译，安装这块才公平；含 C 的包涉及 binary vs 源码编译，会把对比变得不公平（pak 用 binary 免编译）。
2. 快在"安装之外"的开销：无 R 启动、暖缓存、原生求解。`R CMD INSTALL` 本身的 ~1.5s 两边共担、打平。
3. `$PKG` 后接多字节的全角 `（` 时，bash 3.2 可能把字节并进变量名导致"未绑定"；`${PKG}` 用花括号显式界定变量名，避免歧义。
4. 科研诚信：不夸大。明确"哪种场景 pak 更快、为什么"，并把它列为 uvr 的待补齐项，读者才不会被片面数字误导。
