# Step 21：诚实地 benchmark（uvr vs pak）

> 模块：G Benchmark ｜ 对应提交：本步提交（commit 信息含 `Lesson: docs/lessons/step-21-benchmark.md`）｜ 产物：`scripts/bench.sh` + `BENCHMARK.md` ｜ 上一步：[Step 20](step-20-install-command.md)

## 0. 一句话目标
写一个**自计时**脚本，诚实对比 uvr（原生）与 pak（在 R 里）解析依赖的延迟，并把方法论讲清楚。

## 1. 前置回顾
uvr 的功能齐了：解析、求解、lockfile、联网、安装。现在该回答"它到底快不快"。但 benchmark 最容易翻车的不是技术、而是**诚实**——比错了轴会得出误导性结论。

## 2. "测试"：benchmark 脚本本身要能跑出真实数字
benchmark 没有传统单测；它的"绿"就是脚本能稳定跑出可复现的数字。我们把它写成 `scripts/bench.sh`，入库、可复现。

## 3. 实现到通过（真实数字）
关键设计（`scripts/bench.sh`）：
- **release 构建**：`cargo build --release`（debug 跑分无意义）。
- **亚毫秒计时**：macOS 的 `date` 不支持纳秒，故用 perl `Time::HiRes`；并把毫秒写到**独立文件**，命令自身的 stdout/stderr 全丢弃，避免污染计时值：
  ```bash
  MSFILE="$f" perl -MTime::HiRes=time -e \
    'open(my $g,">",$ENV{MSFILE}); my $s=time; system(@ARGV); printf $g "%.1f",(time-$s)*1000;' \
    -- "$@" >/dev/null 2>&1
  ```
- **跑 N 次取最小值**：`sort -n | head -1`（最小值抗噪声）。
- **预热**：先让 pak 加载一次元数据缓存（不计时）。

跑出来（min of 10）：

| 工具 | min ms |
|---|---|
| uvr（原生 release） | 5.4 |
| pak（在 R 里） | 5198.1 |
| 仅 R 启动 | 256.1 |

## 4. 改了哪些文件 / 加了什么
- 新增 `scripts/bench.sh`：自计时对照脚本。
- 新增 `BENCHMARK.md`：结果 + 方法 + 诚实解读。

## 5. 学到的语法 / 技巧
- **bash 函数 + 数组**：`vals+=("$(...)")` 收集每次耗时；`printf '%s\n' "${vals[@]}" | sort -n | head -1` 取最小。
- **perl `Time::HiRes`** 做亚毫秒计时（绕开 macOS `date` 不支持 `%N`）。
- **把计时值与被测命令的输出隔离**：perl 写独立文件、命令输出重定向到 `/dev/null`。
- **`cargo build --release`**：优化构建，唯一适合跑分的版本。

## 6. 方法论上的"设计巧思"
- **min vs mean**：取最小值而非平均——最小值代表"无干扰时的真实下限"，对系统噪声最不敏感。
- **release-only**：debug 版未优化，跑分会假性偏慢、结论失真。
- **预热**：把一次性的缓存加载排除在计时外，量的是"稳态"成本。
- **隔离输出**：被测命令的打印绝不能混进计时数字（我们第一次就踩了这个坑：uvr 的 lockfile 输出污染了计时值，修正为写独立文件）。

## 7. 领域知识
- **pak 的架构**：活在 R 里，有一个"元数据数据库"，每个全新进程都要加载（数秒）。它为"R 会话内反复使用"而设计。
- **R 启动开销**：`Rscript -e 1` 就要 ~256 ms——任何"在 R 里跑的命令行工具"都先天背着这笔账。
- **安装是物理下限**：真正装包由 `R CMD INSTALL` / 编译器决定，与用什么工具调度无关 → uvr 与 pak 在这条轴上**打平**。

## 8. 软件设计理念（也是科研诚信）
- **只比能公平比的轴**：我们比"一次性调用的解析延迟"（uvr 结构上可能赢），不比"安装"（物理下限、打平）。
- **诚实报告**：赢的说赢、平的说平，并讲清"快从哪来"（结构性：无 R 启动、无每进程重载、做得更少），不夸大成"算法更强"。
- **可复现**：脚本入库，任何人可重跑。这正是 benchmark 的可信之本——也呼应了我们造 uvr 想给 R 带来的"可复现"。

## 9. 小测验（自测）
1. 为什么跑分必须用 `--release`？
2. 为什么取最小值而不是平均值？
3. uvr 比 pak 快近千倍，主要是因为"算法更聪明"吗？真正的原因是什么？
4. 为什么"安装"这条轴上 uvr 和 pak 打平？

## 10. 参考答案
1. debug 版未优化、运行慢，跑分会假性偏慢、得出失真结论；release 才反映真实性能。
2. 最小值代表无系统干扰时的真实下限，对调度抖动、后台噪声最不敏感；平均值会被偶发的慢样本拉高。
3. **不是**。主要是结构性原因：uvr 是原生二进制（无 ~256ms 的 R 启动）、不必每进程重载元数据库、且功能上做得比 pak 少。pak 功能远更全面，且为会话内复用而设计。
4. 因为实际构建/安装由 `R CMD INSTALL` 与编译器决定，两者都委托给同一套 R 工具链，调度者是谁都一样——这是物理下限。
