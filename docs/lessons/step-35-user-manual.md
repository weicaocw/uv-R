# Step 35：写一份中英对照用户手册

> 模块：N 文档 ｜ 对应提交：本步提交（commit 信息含 `Lesson: docs/lessons/step-35-user-manual.md`）｜ 产物：`docs/MANUAL.md` + README 刷新 + CHANGELOG/Cargo 版本到 v0.7 ｜ 上一步：[Step 34](step-34-install-with-resolved-r.md)

## 0. 一句话目标
给 uvr 写一份**面向使用者**的完整参考手册（中英对照），并把项目版本推到 v0.7。

## 1. 前置回顾
到这里，uvr 功能已经成体系：lock / install / 多仓库 / 缓存 / R 版本管理。但我们一路写的是 `docs/lessons/`——那是**教学**文档（教你怎么**造**）。使用者要的是另一种文档：怎么**用**。本步补上这块拼图。

## 2. "测试"：文档的"绿"是什么
文档没有 `cargo test`，但仍能定义"通过"：
- **可照着做**：每条命令能复制粘贴跑通（我把示例都按真实输出对齐）。
- **可导航**：有目录、章节、交叉链接。
- **覆盖全**：每个命令、每个产物目录、常见错误都讲到。
- **链接不烂**：相对链接（`../BENCHMARK.md`、`docs/MANUAL.md`）指向真实文件。

## 3. 实现到通过
新建 `docs/MANUAL.md`，分十节：简介 / 安装构建 / 快速上手 / 命令参考 / R 版本管理 / 缓存模型 / 项目布局 / 与 pak 对比 / 排错 FAQ / 设计与边界。每节**中文段 → English 段**对照。

## 5. 学到的技巧（文档工程）
- **教学文档 ≠ 用户文档**：前者讲"为什么这么造、每行什么意思"（`lessons/`）；后者讲"给定任务怎么用最快"（`MANUAL.md`）。读者不同、结构不同，别混在一篇里。
- **中英对照的排版**：本项目统一"中文段在前、English 段在后"，章节标题双语。好处是任一语言读者都能顺着读，而不必左右横跳。
- **示例即测试**：手册里的命令尽量贴真实输出（如 `→ 使用 R / using R 4.5.2`），读者照着跑能对上，文档就不会"说一套做一套"。
- **相对链接**：`docs/MANUAL.md` 里指向 `../BENCHMARK.md`；README 指向 `docs/MANUAL.md`。用相对路径，仓库被 clone / 在 GitHub 上浏览都不断。
- **语义化版本**：功能性新增（R 版本管理）→ 次版本号 +1（0.6 → 0.7）。`Cargo.toml`、CHANGELOG、文档、git tag 四处对齐。

## 6. 设计巧思 / 方法论
- **文档分层**：`README`（30 秒了解 + 入口）→ `MANUAL`（完整用法）→ `lessons/`（从零学造）→ `CHANGELOG`（版本演进）。每层服务不同停留时长的读者。这本身就是**渐进式披露**的设计——和软件 API 一样，先给最常用的，细节按需深入。
- **以发布收尾模块**：一个面向用户的能力（R 版本管理）做完，就配齐"用户能看懂、能上手"的文档再发版。功能与文档同批交付，避免"代码先行、文档欠债"。

## 7. 领域知识
- **R 用户的文档习惯**：R 社区重 vignette（随包的长文档）与 `?help`。uvr 是命令行工具，更接近 CLI 工具的 man page / `--help` 心智；我们用一份 `MANUAL.md` 承载它。
- **`.gitignore` 的建议要写进手册**：`r-lib/`、`.uvr-cache/` 是机器生成物，手册明确建议忽略，免得用户把它们误提交进库。

## 8. 软件设计理念
- **可用性也是功能**：再强的能力，用户找不到、看不懂用法，等于没有。一份好手册的边际价值，常常不亚于一个新功能。
- **诚实写边界**：手册第 10 节明确列出"装的是源码包、binary 未做、并行未做"。把局限写在明处，比让用户踩坑后自己发现更可信。

## 9. 小测验（自测）
1. `docs/lessons/` 和 `docs/MANUAL.md` 面向的读者与目的有何不同？
2. 为什么功能新增让版本从 0.6 升到 0.7，而不是 0.6.1？
3. 手册里为什么要专门写一节"设计与边界"，把没做的功能也列出来？
4. 示例命令尽量贴真实输出，对文档质量有什么实际意义？

## 10. 参考答案
1. `lessons/` 是**教学**：教零基础读者**如何从头构建**，讲语法 / 设计 / 领域知识，按步骤推进。`MANUAL.md` 是**参考**：给已经要用工具的人**最快完成任务**的命令与说明，按主题组织。
2. 语义化版本：向后兼容的**功能新增**升次版本号（minor，0.6→0.7）；0.6.1 是补丁号，留给只修 bug、无新功能的发布。
3. 诚实与预期管理：让用户提前知道"含编译大包目前不如 pak、binary 未做"，避免误用和误解；把已知局限摊开，也是给后续工作留的路标。
4. 文档与实现一致、可复现。读者复制命令能得到手册所示结果，建立信任；若示例与真实输出脱节，文档就会误导，反而有害。
