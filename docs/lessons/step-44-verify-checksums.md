# Step 44：下载后校验完整性——发现篡改就拒装

> 模块：R 完整性 ｜ 对应提交：本步提交（commit 信息含 `Lesson: docs/lessons/step-44-verify-checksums.md`）｜ 产物：`Cargo.toml`（`sha2`）+ `install.rs`（`verify_hash`）+ `commands.rs`（安装前校验）｜ 上一步：[Step 43](step-43-record-checksums.md)

## 0. 一句话目标
下载完每个 tarball 后，算出它的 SHA256 与锁文件里记的比对；**不符就报错、拒绝安装**。

## 1. 前置回顾
Step 43 把仓库声明的校验和记进了 lockfile（v3）。但记下来不用等于没用。本步真正去**验证**：拿到的字节是不是仓库登记的那一份。

## 2. "测试"先行（TDD 红）
纯校验函数：
```rust
verify_hash(文件, "sha256:<正确值>") == Ok
verify_hash(文件, "sha256:0000")     == Err   // 不符
verify_hash(文件, "")                == Ok    // 空 → 跳过
verify_hash(文件, "md5:...")         == Ok    // 非 sha256 → 暂跳过
```
端到端（本机演示）：
```
$ uvr sync --lib ./r-lib            # 校验 SHA256 通过 → 安装
synced dotenv 1.0.3.9000

# 篡改缓存里的 tarball 后再 sync：
$ echo TAMPERED > .uvr-cache/tarballs/dotenv_1.0.3.9000.tar.gz
$ uvr sync --lib ./r-lib
同步失败 / sync failed: dotenv 1.0.3.9000: 校验和不符 / checksum mismatch:
  期望 941d…ded11, 实得 314c…17b2          # ← 篡改被发现，拒装
```

## 3. 实现到通过（TDD 绿）
- `Cargo.toml`：加依赖 `sha2 = "0.10"`（RustCrypto，纯 Rust）。
- `install.rs`：`verify_hash(file, expected)`——`expected` 是 `sha256:<hex>` 才校验：读文件、`Sha256::digest`、转十六进制、不区分大小写比对；为空或 `md5:` 则放行（MD5 校验暂未实现）。
- `commands.rs`：`run_plan` 阶段二在 `install_tarball` **之前**插入 `verify_hash(&tarball, &item.hash)?`，并把出错信息带上包名。
69 测试全绿；篡改演示如上被拦下。

## 4. 改了哪些文件 / 加了什么
- `Cargo.toml`：`sha2` 依赖。
- `src/install.rs`：`verify_hash` + 1 个测试（正确/错误/空/非 sha256）。
- `src/commands.rs`：安装前校验、错误带包名。

## 5. 学到的语法 / 技巧
- **加一个 crate**：`Cargo.toml` 的 `[dependencies]` 写一行，`cargo build` 自动拉取编译。这正是 Rust 生态的力量——不自己实现 SHA256（易错），用经审计的 `sha2`。（注意：这是**项目本地**依赖，进 `Cargo.lock`，不污染系统——与本机"绝不污染全局环境"的底线一致。）
- **`strip_prefix` + let-else 守卫**：`let Some(want) = expected.strip_prefix("sha256:") else { return Ok(()) }`——同时完成"是不是 sha256""取出 hex""不是就跳过"三件事，优雅。
- **`Sha256::digest(bytes)`**：一次性算哈希；返回固定长度字节数组，`.iter().map(|b| format!("{b:02x}"))` 转成十六进制串（`{:02x}` = 两位小写十六进制、不足补 0）。
- **`eq_ignore_ascii_case`**：十六进制大小写无所谓，用它比对更稳健。
- **校验在安装前、错误带上下文**：`verify_hash(...).map_err(|e| format!("{name} {version}: {e}"))?`——失败信息说清"哪个包"出了问题。

## 6. 设计巧思 / 方法论
- **fail-closed（出错即停）**：校验不过就**中止整次安装**，绝不"先装了再说"。安全检查的正确姿势是"有疑即拒"，而非"尽量继续"。
- **优雅降级、不假装**：仓库没给 sha256（空 / 只有 md5）时**放行**而非报错——我们不假装能验证拿不到的东西。把"暂不支持 MD5 校验"写进文档，诚实标注能力边界（见模块 J 的同款态度）。
- **校验在"安装前"这个位置**：下载（可能并行、可能命中缓存）之后、`R CMD INSTALL`（有副作用、改本地库）之前。**在产生副作用前设最后一道闸**。

## 7. 领域知识（供应链安全）
- **完整性 vs 真实性**：SHA256 比对保证"内容 = 仓库登记的内容"（完整性，防损坏 / 防篡改）。它**不**保证"仓库本身可信"（真实性，那要签名 / TLS 证书）。uvr 这一步做的是完整性，是供应链防护的第一层。
- **为什么对包管理器是刚需**：你在 CI / 新机器上 `sync` 时，tarball 经过网络、镜像、缓存层层流转，任一环节损坏或被替换都可能让你装上**非预期的代码**。下载后校验是抵御这类风险的标准做法（`cargo`、`pip --require-hashes`、`npm` integrity 皆然）。
- **锁版本 + 锁校验和 = 完整可复现**：版本防"装到别的版本"，校验和防"装到被改过的同名同版本"。两者齐全，`uvr sync` 才是真正可信的"按图施工"。

## 8. 软件设计理念
- **复用胜过重造**：SHA256 自己写既难又危险（密码学实现差之毫厘谬以千里）。引入经审计的 `sha2`，把正确性外包给专业实现——**知道什么该自己做、什么该交给生态**。
- **安全是默认、不是选项**：校验自动发生，用户无需开关。把正确的安全行为做成**默认且无感**，才能真正保护到大多数人。

## 9. 小测验（自测）
1. 为什么校验放在"下载之后、`R CMD INSTALL` 之前"，而不是别的位置？
2. 仓库没提供 sha256（只有 md5 或啥都没有）时，`verify_hash` 为什么选择放行而不是报错？
3. SHA256 校验保证了什么、又**不**保证什么？
4. 为什么用 `sha2` 这个 crate 而不自己实现 SHA256？这与本机"不污染环境"的底线冲突吗？

## 10. 参考答案
1. 下载后才有文件可算哈希；安装会改本地库（副作用）。把校验放在"有副作用之前的最后一刻"，一旦不符就能在污染环境前中止——这是设最后一道闸的最佳位置。
2. 我们只能验证仓库**提供了**的校验和；拿不到 sha256 就无从验证，此时报错会让"仓库恰好没给"的正常情况也装不了。选择放行 + 文档标注"MD5 校验未实现"，是诚实的优雅降级，不假装有不具备的能力。
3. 保证**完整性**：拿到的字节 = 仓库登记的字节（防传输损坏 / 防篡改）。不保证**真实性**：仓库本身是否可信、是否被冒充——那需要签名 / 证书等另一层机制。
4. 密码学实现极易出微妙错误，自写风险高；用经审计的 `sha2` 更可靠。不冲突：Cargo 依赖是**项目本地**的（进 `Cargo.lock`、装在项目 target 里），不碰系统 / 全局环境——正是本机底线认可的方式（被禁的是污染系统 Python/Node，而非项目内的语言包管理）。
