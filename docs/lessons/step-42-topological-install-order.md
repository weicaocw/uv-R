# Step 42：拓扑排序——让依赖先于依赖者安装

> 模块：Q 拓扑序安装 ｜ 对应提交：本步提交（commit 信息含 `Lesson: docs/lessons/step-42-topological-install-order.md`）｜ 产物：`src/commands.rs`（`topo_order` + 接入 `install_plan`/`sync_plan`）｜ 上一步：[Step 41](step-41-self-contained-sync.md)

## 0. 一句话目标
安装一组包前，先按**依赖关系**排序：被依赖的在前、依赖别人的在后（拓扑排序）。

## 1. 前置回顾
此前 `install` / `sync` 按**包名字母序**（`BTreeMap` 顺序）逐个装。但 `R CMD INSTALL` 装一个包时，常需要它的依赖**已经装好**（构建期要加载）。字母序不保证这点：若 `zoo` 依赖 `abc`，字母序恰好对；但 `abc` 依赖 `zoo` 就会先装 `abc`、失败。正确的顺序是**拓扑序**——这是图论里的经典问题。

## 2. "测试"先行（TDD 红）
用三种依赖图钉死行为：
```
链：C→B→A      输入乱序 [C,A,B]      → 输出 [A,B,C]
菱形：D→B,C；B,C→A  输入 [D,C,B,A]  → 输出 [A,B,C,D]（B/C 名字序）
独立：A,B 无依赖                      → [A,B]（名字序）
空图（自包含 sync 无 --repo）          → 名字序，绝不丢包
```
本机：`cargo test topo` → **4 passed**。

## 3. 实现到通过（TDD 绿）——Kahn 算法
拓扑排序的经典做法（Kahn）：
1. 给每个包算**入度** = 它在本批集合内、尚未安装的依赖数。
2. 把入度为 0 的（依赖都齐了）放进"就绪集合"。
3. 反复：从就绪集合取一个装上 → 它的**依赖者**入度各减 1 → 减到 0 的进就绪集合。
4. 直到取完。若还有剩（成环），按名字补上、绝不丢。

确定性关键：就绪集合用 `BTreeSet`，每次取**名字最小**的——同样的输入永远得到同样的输出。
边只取**指向本批集合内**的依赖（`names.contains`），自动忽略 base 包（如 `R`）和不在本次安装里的包。`install_plan` / `sync_plan` 末尾调用 `topo_order(items, &index)`。66 测试全绿。

## 5. 学到的语法 / 技巧
- **`BTreeMap` / `BTreeSet` 给确定性**：哈希容器遍历顺序不定；有序容器按 key 排序遍历。算法要可复现，就用有序容器——这和 lockfile 用 `BTreeMap` 保证确定输出是同一招。
- **`entry(k).or_default().push(v)`**：构建"邻接表"（dep → 依赖它的人）的惯用法：键不存在就插一个默认空 `Vec` 再 push。一行建图。
- **`while let Some(n) = ready.iter().next().cloned()`**：从有序集合反复取最小元素；`.next()` 取首个（最小），`.cloned()` 拿到拥有的 `String` 再从集合移除，避免边遍历边改的借用冲突。
- **`filter_map(|n| by_name.remove(&n))`**：按拓扑序的名字列表，从"名字→项"的 map 里**取走**对应项，组装成最终有序的 `Vec`。`remove` 返回 `Option`，`filter_map` 顺手解包。
- **边界条件 `**v == 0`**：Rust 2024 对"隐式借用里再解引用"的模式更严格，`filter(|(_, v)| **v == 0)` 显式解两层引用（`&&usize`）。（这一步真踩了编译错误并修掉。）

## 6. 设计巧思 / 方法论
- **优雅降级**：信息不足（自包含 `sync` 没传 `--repo`，索引为空）时，没有边 → 退化为名字序，**绝不报错、绝不丢包**。算法对"不完整输入"也给合理结果，这比"缺信息就崩"健壮得多。
- **成环兜底**：求解后理论上无环，但代码仍处理"万一成环"——把剩余项补在后面。**永不丢数据**是底线，哪怕前提被破坏也优雅退出。
- **纯函数 + 确定性**：`topo_order` 只依赖入参，同输入同输出，能被穷举单测。又一次"把核心逻辑做成纯函数"。

## 7. 领域知识（R / 包管理 + 计算机科学）
- **为什么装序重要**：`R CMD INSTALL` 在"字节编译 / 准备懒加载"阶段会**加载**包的依赖；依赖没装好就失败。所以必须依赖先行。`pak` / `renv` 等都做依赖图调度。
- **拓扑排序是包管理的核心算法**：从 `make`、`apt`、`cargo` 到 `npm`，"按依赖顺序处理"无处不在，背后都是拓扑排序。学会它，你就懂了一大类构建 / 安装工具的骨架。
- **它也是"多包并行安装"的前置**：知道了依赖层次，才能"同一层的包并行装、层间按序"。本步排好序，为将来的并行安装铺好了路。

## 8. 软件设计理念
- **正确性的投资**：字母序"大多数时候能用"，但拓扑序"总是对"。把"碰巧能用"升级成"保证正确"，是工程成熟度的体现——即使在本机一时难以构造多依赖纯 R 包来端到端演示，单测已严格证明排序逻辑。
- **算法封装**：把图论塞进一个边界清晰的 `topo_order`，调用方只看到"给一批项、还回有序的一批"。复杂度被封装，接口保持朴素。

## 9. 小测验（自测）
1. 为什么按包名字母序安装可能失败，而拓扑序总是安全？
2. Kahn 算法里"入度"指什么？什么时候一个包变得"就绪"？
3. 就绪集合为何用 `BTreeSet` 而非 `HashSet`？
4. 自包含 `sync`（没传 `--repo`、索引为空）时，`topo_order` 会怎样？为什么这样设计是合理的？

## 10. 参考答案
1. `R CMD INSTALL` 装包时需其依赖已装好；字母序与依赖关系无关，可能把依赖者排在依赖前面导致失败。拓扑序保证"每个包安装时，它在本批内的依赖都已装"，故总是安全。
2. 入度 = 该包在本批集合内、**尚未安装**的依赖数量。当它的入度降到 0（所有依赖都装好了），它就"就绪"，可以安装。
3. `BTreeSet` 有序，每次取名字最小者，使输出**确定可复现**（同输入同输出）；`HashSet` 遍历顺序不定，会让安装顺序在不同运行间漂移，破坏可复现性。
4. 索引为空 → 找不到任何依赖边 → 所有包入度 0 → 按名字序全部输出，绝不丢包。合理：v2 自包含 sync 本就没有依赖图信息，退化为名字序是"尽力而为"，而非报错——保持了功能可用。
