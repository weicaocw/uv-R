# Benchmark: uvr vs pak

> **诚实第一。** 多轴对比，并对每条轴说清"快从哪来、哪里打平、哪里还不公平"。
> 复现：`scripts/bench.sh [package] [N]`（本机：macOS x86_64，R 4.5.2，pak 0.9.5）。

## 结果（本机，`dotenv` 纯 R 包）

### 1) 解析依赖的调用延迟（min of N，一次性进程）
| 工具 | ms |
|---|---|
| **uvr**（原生 release） | **6.0** |
| pak（在 R 里） | 5415 |
| 仅 R 启动（`Rscript -e 1`） | 267 |

→ ~900×。**结构性**：pak 每次冷进程要付 R 启动（267ms）+ 重载元数据库（数秒）；uvr 是原生二进制、只读小 `PACKAGES`。

### 2) 元数据 `lock`（`uvr lock --repo`，冷抓 vs 暖缓存）
| | ms |
|---|---|
| cold（联网抓） | 165 |
| **warm（暖缓存）** | **6.0** |

→ ~27×。暖缓存免去网络往返——uv 的招牌轴。

### 3) 端到端安装 `dotenv`（fetch + resolve + download + install，装进临时库）
| | ms (single) |
|---|---|
| uvr cold（清缓存） | 2080 |
| **uvr warm（暖缓存）** | **1575** |
| pak warm | 5478 |

→ 即便 uvr **冷** vs pak **暖**，uvr 也快 ~2.6×；warm vs warm ~3.5×。

## 诚实解读
- **uvr 的优势是结构性的**，不是"算法更聪明"：原生二进制（无 R 启动）、不每进程重载元数据库、暖缓存、求解走 `pubgrub`（与 pak 同级的算法）。
- **共享地板（打平的部分）**：实际 `R CMD INSTALL` 一个包 ~1.5s，两边都得付（uvr warm 的 1575ms 几乎全是这个）。uvr 赢在**安装之外的开销**（元数据、求解、进程启动），而不是装得更快。
- **公平性范围**：本对比用**纯 R 包**（`dotenv`，无编译）——两边都不编译，公平。对**含 C/C++/Fortran 的包**，pak 会用预编译 **binary**（免编译），而 uvr 目前装**源码**（要编译）；那种场景 pak 的安装会更快——直到 uvr 支持 binary 包。
- **环境说明**：本机 R 默认 `pkgType="source"`（`contrib.url(type="binary")` 回退到源码路径），r-universe 的 macOS binary 路径对测试包 404；故 binary 支持在此环境难以干净演示，列为未来工作。

## 还差什么才算"全面对齐 pak"
- **binary 包**（模块 J，本环境受限）：让含编译的包免编译安装。
- **并行下载 / 安装**（模块 K）：装很多包时的吞吐。
- 这两项补齐后，连"安装大量含编译的包"这种 pak 的主场也能公平对比。

## 结论
uvr 现在已能与 pak 做**端到端、可复现、诚实**的对比，并在 **解析 / 锁定 / 暖缓存 / 纯 R 包端到端安装** 这些轴上**全面更快**——快在结构（原生、无 R 启动、暖缓存），而非取巧；`R CMD INSTALL` 这块物理地板诚实打平。对"含编译的大包"，待 binary + 并行补齐后再全面对齐。
