# Benchmark: uvr vs pak

> **诚实第一。** 这里只比"能公平比、且 uvr 结构上可能赢"的那条轴——**一次性命令调用里"解析依赖"的延迟**。
> 真正的重活（下载、`R CMD INSTALL` / 编译）两者都得做、且都委托给同一套 R 工具链，是**打平**的，故不在此比较。
> 复现：`scripts/bench.sh [package] [N]`。

## 结果（本机，min of 10 runs）
package = `dotenv`（纯 R、零依赖）

| 工具 | 调用延迟 (min, ms) |
|---|---|
| **uvr**（原生 release 二进制） | **5.4** |
| pak（在 R 里 `pak::pkg_deps`） | 5198.1 |
| 仅 R 启动（`Rscript -e 1`） | 256.1 |

→ 一次性调用下，uvr 比 pak 快约 **960×**。

## 方法
- 自写计时（`scripts/bench.sh`）：用 perl `Time::HiRes` 取亚毫秒，每项跑 10 次取**最小值**（最小值最稳、受系统噪声影响最小）。**不用 hyperfine**（避免 brew 安装）。
- uvr 用 **release** 构建（debug 跑分无意义）。
- pak 先**预热**元数据缓存（一次不计时的调用）再计时。
- uvr 离线解析本地 `PACKAGES`（已抓取的真实 r-universe 索引）；pak 用它自己的元数据库。

## 诚实解读：快从哪来、又意味着什么
uvr 的优势**主要是结构性的**，不是"算法更聪明"：
1. **没有 R 子进程**：pak 活在 R 里，每次命令行调用都要付 ~256 ms 的 R 启动；uvr 是原生二进制，启动 ~ms。
2. **没有每进程重载元数据库**：pak 每个全新进程都要"Loading metadata database"（数秒）；uvr 只读一个小 `PACKAGES`。
3. **uvr 做得更少**：pak 还处理 `SystemRequirements`、多仓库、binary 选择、并行安装等——功能远比 uvr 全面。

所以这个数字是 uvr 的**主场**（一次性 CLI 调用）、pak 的**最坏情形**。公平地补充：
- pak 为"在一个 R 会话里反复使用"而设计，元数据每会话只加载一次；若在同一 R 会话里连续解析，pak 每次的成本远低于 5.2 s。
- **安装才是真正的重活**，由 `R CMD INSTALL` / 编译器决定，uvr 与 pak **打平**（都委托给同一套 R 工具链）——这条我们不吹。

## 结论
uvr 的价值正如预期：**一个零依赖、即时启动的原生工具**，在"解析 / 锁定 / 一次性调用"这类轻量高频操作上有**数量级**优势；在"实际安装"这类重活上与 pak **持平**。这与 uv 之于 Python 的定位一致：**赢在启动与轻量操作，重活诚实持平**。
